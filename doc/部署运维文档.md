# 部署运维文档

## 1. 生产配置

Linux 宝塔安装 Nginx、Go 运行环境和 MySQL 5.7。后端复制 `backend/.env.example` 为 `.env`，设置远程 MySQL DSN、随机 `JWT_SECRET`、独立 `DATA_ENCRYPTION_SECRET`、`PUBLIC_BASE_URL`、`ADMIN_TOTP_SECRET` 和支付/AI 配置。`.env` 严禁提交 Git。

## 2. 发布步骤

```bash
cd backend
go run ./cmd/migrate
go build -o tiku-zong-api ./cmd/server
./tiku-zong-api

cd ../frontend
pnpm install --frozen-lockfile
pnpm run build
```

Nginx 将前端静态目录指向 `frontend/dist`，将 `/api/` 反向代理到 Go 服务；配置 HTTPS、HTTP 到 HTTPS 跳转和合理的上传/请求体限制。可从 `deploy/nginx/tiku-zong.conf` 复制模板到宝塔站点配置：替换域名、前端根目录和监控 IP，并将同一规则应用到 HTTPS 的 `443` server 块。模板已设置 12 MiB 上传上限、`/payment/result` 的 SPA 回退、75 秒 AI 请求代理超时和默认禁止公网访问 `/metrics`。

生产环境的 `PUBLIC_BASE_URL` 必须是 Nginx 对外 HTTPS 地址（例如 `https://tiku.example.com`），不能写内部 `:8088` 地址；支付收银台的通知和回跳地址均由该值生成。

若不使用宝塔进程守护，可使用 `deploy/systemd/tiku-zong-api.service`：将二进制部署到 `/opt/tiku-zong/backend/tiku-zong-api`、生产环境变量保存到权限为 `600` 的 `/etc/tiku-zong/tiku-zong.env`，再执行：

```bash
sudo cp deploy/systemd/tiku-zong-api.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now tiku-zong-api
sudo systemctl status tiku-zong-api
```

## 3. 初始化与维护

使用一次性环境变量 `ADMIN_EMAIL`、`ADMIN_PASSWORD` 执行 `go run ./cmd/admin` 初始化管理员。服务每分钟自动关闭过期支付订单，也保留管理员手动接口。存活检查使用 `/healthz`，依赖就绪检查使用 `/readyz`。

## 4. 备份与恢复

每日使用 `mysqldump --single-transaction --routines --events` 备份数据库，保留至少 7 天并存储到独立位置；每周执行恢复演练。备份文件同样视为敏感数据，不放在 Web 根目录。

仓库提供 `backend/scripts/backup-mysql.sh` 和 `backend/scripts/restore-mysql.sh`。两个脚本只从环境变量读取数据库凭据，不读取或输出 `.env`、DSN、密码和密钥。首次部署先使用最小权限的备份账户创建受限权限的 `/etc/tiku-zong/backup.env`（权限 `600`），其中包含 `MYSQL_HOST`、`MYSQL_PORT`、`MYSQL_USER`、`MYSQL_PASSWORD`、`MYSQL_DATABASE`、`BACKUP_DIR=/var/backups/tiku-zong`、`BACKUP_RETENTION_DAYS=7`。

```bash
chmod 700 backend/scripts/backup-mysql.sh backend/scripts/restore-mysql.sh
set -a; source /etc/tiku-zong/backup.env; set +a
backend/scripts/backup-mysql.sh
```

宝塔计划任务可每日执行上述备份命令。恢复演练必须在隔离数据库进行，并显式提供待恢复文件和确认值；脚本会检查 SHA-256 校验文件（若存在）并要求 `RESTORE_CONFIRM` 与目标数据库完全相同：

```bash
export BACKUP_FILE=/var/backups/tiku-zong/tiku_20260714T000000Z.sql.gz
export RESTORE_CONFIRM=tiku_restore_drill
export MYSQL_DATABASE=tiku_restore_drill
backend/scripts/restore-mysql.sh
```

恢复生产库前必须先停止写入服务、完成独立备份并由管理员复核。脚本不会删除旧备份；保留策略仅清理由备份脚本创建且超过天数的同名压缩备份与校验文件。

恢复脚本默认拒绝导入到非空数据库，避免误覆盖。仅在完成备份、停止写入和人工复核后，才可显式设置 `RESTORE_ALLOW_NONEMPTY=1` 进行非空库恢复；日常演练应始终使用新建的隔离数据库。

## 5. 故障排查

先检查 `/healthz`、服务日志、MySQL 连接、迁移版本、Nginx 代理和环境变量；支付回调重点检查签名、金额和订单状态，AI/OCS 重点检查超时、源优先级和密钥配置。

Nginx 变更后执行 `nginx -t`，再在宝塔中平滑重载。支付回跳出现 404 时，检查 Nginx 的 `location /` 是否保留 `try_files ... /index.html`，并确认 `PUBLIC_BASE_URL` 指向前端同域 HTTPS 地址。

## 8. 监控指标

`/metrics` 输出 Prometheus 文本格式的请求总数、AI 队列深度/容量/Worker 数和 `tiku_database_ready`（MySQL 就绪为 `1`、不可用或未配置为 `0`）。该端点不包含密钥和业务内容，但生产环境仍应通过 Nginx、内网监听或访问控制限制来源。

仓库提供 `deploy/prometheus/prometheus.yml` 与 `deploy/prometheus/tiku-zong-alerts.yml`。Prometheus 应从同机 `127.0.0.1:8088/metrics` 抓取，不需要开放公网指标端点；将告警规则复制到 `/etc/prometheus/rules/` 后执行 `promtool check config`、`promtool check rules` 并重载 Prometheus。模板覆盖 API 不可抓取、MySQL 未就绪、AI 队列持续超过 80% 和 AI Worker 为 0 四类告警。

## 6. Redis 限流

设置 `REDIS_ADDR` 后，搜题接口使用 Redis `EVAL` 原子递增并按分钟窗口过期，适用于多实例部署；Redis 不可用时自动回退到当前进程内存计数。Redis 应限制网络访问并配置独立凭据或 ACL。

## 7. AI 调用队列

AI 兜底请求通过进程内有界队列调度，HTTP 接口仍同步返回，避免突发请求无限创建网络调用。使用 `AI_QUEUE_SIZE` 设置待处理队列容量，使用 `AI_QUEUE_WORKERS` 设置并发 worker 数；队列满时返回失败且不扣费。多实例部署时应分别评估每个实例的队列容量，并配合 Redis 限流和上游 AI 配额。
