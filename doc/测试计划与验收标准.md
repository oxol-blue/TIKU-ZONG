# 测试计划与验收标准

## 1. 自动化测试

后端在 `backend/` 执行 `go test ./...` 和 `go vet ./...`；前端在 `frontend/` 执行 `pnpm run type:check` 和 `pnpm run build`。新增业务规则必须提供 Go 单元测试，外部支付、OCS、AI 使用本地 mock 验证。

仓库的 `.github/workflows/ci.yml` 会在推送到 `main` 和向 `main` 提交 Pull Request 时执行同一组门禁：Go 1.26 的测试与静态检查，以及 Node 22 + pnpm 10 的前端依赖安装、类型检查和生产构建。GitHub 远端可用后，应将 CI 通过作为合并条件。

## 2. 核心验收链路

注册 → 登录 → 获取 API Key → 创建/支付套餐订单 → 异步回调 → API 搜题 → 本地/OCS/AI 返回文字答案 → 正确扣费 → 调用日志 → 答案反馈。

## 3. 必测安全项

验证登录失败锁定、验证码单次使用、API Key 哈希与禁用、管理员角色隔离、TOTP、IP 黑白名单、API 防刷、AI/支付密钥不回显、OCS 不执行任意 JavaScript。

## 4. 数据规则

覆盖题目哈希去重、多选 `###`、简答多答案、题目/选项相似匹配、OCS 优先级/多数投票、AI 缓存、AI 失败不扣费、套餐到期、叠加、优惠券核销、重复支付回调、部分退款和订单超时。

## 5. 发布前执行清单

### 自动化门禁

- [ ] `cd backend; $env:GOCACHE="$PWD/.gocache"; go test ./...`
- [ ] `cd backend; $env:GOCACHE="$PWD/.gocache"; go vet ./...`
- [ ] `cd frontend; pnpm.cmd run type:check`
- [ ] `cd frontend; pnpm.cmd run build`
- [ ] `cd backend; go run ./cmd/migrate`，确认迁移版本与发布版本一致

### 业务验收

- [ ] 新用户使用邮箱、密码和邀请码注册，邀请码达到上限后注册失败
- [ ] 创建 API Key，确认数据库只保存哈希；使用 Key 搜题并确认失败请求不扣费
- [ ] 导入含选项和多答案题目，确认答案为选项文字且多选使用 `###`
- [ ] 创建三类套餐，验证叠加、过期优先、普通额度和 AI 额度扣减
- [ ] 使用 mock 支付回调，重复通知只发放一次套餐
- [ ] 使用 mock OCS/AI，验证优先级、多数投票、AI 缓存和 AI 失败不扣费
- [ ] 提交答案反馈，重复提交同一请求被拒绝
- [ ] 用户查看调用记录和反馈记录，确认题目原文不会出现在列表响应中
- [ ] 管理员按用户邮箱/请求 ID/题目哈希/反馈类型筛选反馈，并确认列表不显示题目原文
- [ ] 轮换 API Key 后旧 Key 失效，新 Key 仅在响应中返回一次并可下载 OCS 配置

### 生产安全

- [ ] 配置 HTTPS、独立 JWT/加密密钥、管理员 TOTP 和 Redis ACL
- [ ] 验证 IP 黑白名单、Redis 限流和 Redis 故障回退
- [ ] 完成 MySQL 备份恢复演练，并确认备份不位于 Web 根目录
- [ ] 使用 `backup-mysql.sh` 生成备份并校验 SHA-256；在隔离库使用 `restore-mysql.sh` 恢复后执行迁移和关键查询验证
- [ ] 管理员访问仪表盘，核对用户数、支付金额、调用成功率、AI/OCS 调用量和平均耗时与数据库聚合结果一致
- [ ] 管理员查看订单退款明细，重复提交相同 `refundNo` 不重复累计退款金额
- [ ] 管理员运行订单对账，能识别过期未关闭、已支付无套餐实例和退款金额不一致异常
- [ ] 管理员执行套餐实例补偿，确认仅修复已支付/部分退款的缺失关联，重复执行不重复创建实例
- [ ] GitHub Actions 的 `Backend test and vet` 与 `Frontend type check and build` 两个任务均通过
